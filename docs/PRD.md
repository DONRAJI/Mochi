# 🧊 모찌(Mochi) — 서비스 기획안 (PRD v1.1)

> v1.1(2026-06-30): P0 구현·배포 후 효과·개인화·기록 확장 논의를 **11장**에 보완. (1~10장은 v1.0 그대로)

> **한 줄 정의**: "오늘 뭐 먹지"를 모찌가 대신 풀어주는, **죄책감 없는 사전 제안형 식사 컴패니언**.
> 다이어트를 *관리(tracking)* 가 아니라 *수집(collecting)* 의 즐거움으로 바꾼다.

---

## 0. 이 문서를 읽는 법 (PM 코멘트)

원안의 방향성은 매우 좋습니다. 다만 기획을 다듬으며 **딱 하나의 전략적 결정**을 먼저 내렸습니다.

> **앱의 정체성을 "냉장고 비우기 앱" → "오늘 잘 먹었다는 기록 앱"으로 재정의합니다.**

이유는 명확합니다. 원안의 핵심 가치(결정 피로 해소·죄책감 제로·수집)는 **요리 여부와 무관하게** 성립하는데, 기능 설계가 '냉장고/재료'에 묶이는 순간 비요리 사용자(타겟의 상당수인 1인 가구 여성)가 이탈합니다. 그래서 냉장고는 **앱의 전부가 아니라 입력 방식 중 하나**로 격하시키고, 모든 기능을 **제안 → 기록 → 수집**이라는 3대 루프 위에서 재구성했습니다. 이 결정이 [핵심 과제] 섹션(8장)의 비요리 사용자 포용으로 자연스럽게 연결됩니다.

## 1. 핵심 차별점 (USP)

대부분의 다이어트 앱은 **사후 기록형**입니다. 사용자가 먹은 걸 입력하면 숫자(칼로리)를 보여주고, 초과하면 빨간 경고로 죄책감을 줍니다. 모찌는 이 구조를 **세 가지 축에서 뒤집습니다.**

| 축 | 기존 앱 (예: 칼로리 트래커류) | 모찌 |
|---|---|---|
| **방향** | 사후 기록 → "당신이 뭘 먹었는지 입력하라" | **사전 제안** → "당신이 뭘 먹을지 모찌가 정해준다" |
| **감정** | 숫자·실패·경고 (죄책감 유발) | **죄책감 제로** (칭찬만, 벌 없음, 숫자 숨김) |
| **동기** | 의지력·목표 압박 (쉽게 소진) | **수집·육성 욕구** (포켓몬 도감 메커니즘으로 지속) |

**4가지 USP를 한 문장씩으로:**

1. **결정 대행 (Decision-as-a-Service)** — "오늘 뭐 먹지"라는 매일의 인지 부하를 모찌가 대신 처리한다. 다이어트 실패의 1차 원인은 의지가 아니라 *결정 피로*다.
2. **죄책감 제로 설계** — 강제 칼로리 타겟·실패 메시지·빨강 경고가 없다. 진행도는 숫자가 아니라 **모찌의 성장**으로 보여준다.
3. **수집이 곧 다이어트 (Pokédex Hook)** — 만든 요리·먹은 음식·써본 재료가 전부 도감에 쌓인다. "건강하게 먹는 행위"가 "도감을 채우는 행위"로 치환된다. → **리텐션의 핵심 엔진**
4. **모드 무관 포용** — 요리하든, 배달시키든, 편의점에서 사 먹든 똑같이 제안받고·기록하고·수집한다. (8장 참조)

> **포지셔닝 한 줄**: "MyFitnessPal이 회계사라면, 모찌는 냉장고에 사는 다정한 룸메이트다."

## 2. 타겟 & 페르소나

원안의 타겟(다이어트 초보·여성·가벼운 관리)을 유지하되, **요리 성향**을 핵심 분기축으로 잡아 두 페르소나로 분리합니다. 이 분리가 기능 우선순위와 8장 전략의 근거가 됩니다.

### 페르소나 A — "요리하는 지윤" (1차 타겟)
- 27세, 1인 가구, 주 3~4회 자취 요리. 재료를 남겨서 버리는 게 스트레스.
- **니즈**: 가진 재료로 뭘 해먹을지, 유통기한 임박 재료 활용.
- **앱 사용**: 냉장고 입력 → 레시피 추천 → 요리 도감 수집.

### 페르소나 B — "안 해먹는 수아" (★ 확장 타겟, 8장의 주인공)
- 25세, 1인 가구, 야근 잦음. 배달·편의점·외식 위주. 요리는 거의 안 함.
- **니즈**: 매번 메뉴 고르기 귀찮음("오늘도 또 뭐 시키지"), 그래도 너무 막 먹긴 싫음.
- **앱 사용**: 외식/배달 추천 받기 → 사진으로 기록 → 간편식 도감 수집.
- **왜 중요한가**: 1인 가구 + 다이어트 초보 + 여성 세그먼트에서 **실제로 다수**다. A만 노리면 TAM의 절반을 버린다.

## 3. 메뉴 구조도 (IA)

5탭 구조. **홈을 모찌의 공간으로** 두고, 냉장고·식단·도감을 병렬 배치, 숫자/기록은 마이로 숨깁니다.

```
모찌 (App)
│
├── 🏠 홈 (모찌의 방)           ← 진입 시 첫 화면
│   ├── 모찌 캐릭터 (상태 반영: 기쁨/졸음/대기)
│   ├── 오늘의 제안 카드 (스와이프)
│   ├── 스트릭 위젯 (보호권 포함)
│   └── 빠른 액션 (먹었어요 / 재료 추가)
│
├── 🧊 냉장고 (재료)           ← '입력 모드' 중 하나
│   ├── 재료 그리드 (스티커, 카테고리 탭)
│   ├── 재료 추가 (수동/사진/영수증/바코드)
│   ├── 유통기한 관리
│   └── 선호·비선호·알러지 태그
│
├── 🍽️ 식단 (오늘 뭐 먹지)      ← 추천 엔진의 얼굴
│   ├── [요리 모드] 레시피 추천 + 매칭률
│   ├── [외식/배달 모드] 메뉴 추천 ★(8장)
│   ├── 주간 식단 캘린더 (드래그 재배치)
│   └── 장보기 리스트 (자동 생성)
│
├── 📖 도감 (수집)             ← 리텐션 엔진
│   ├── 요리 도감 / 재료 도감 / 간편식 도감 ★
│   ├── 컴플리트 게이지 (카테고리·시즌별)
│   └── 뱃지·보상함 (스티커/옷/테마)
│
└── 👤 마이 (기록·더보기)       ← 숫자는 전부 여기로
    ├── 체중 기록 (6장 — 모찌 동행 그래프)
    ├── 통계 ('더보기' 안에 숨김)
    └── 설정 / 알림 / 모찌 꾸미기
```

**IA 설계 원칙 3가지**
1. **홈에는 숫자가 없다.** 체중·칼로리·달성률은 전부 마이 탭 '더보기'로 격리 → 초보자 피로도 차단.
2. **'식단' 탭이 모드를 흡수한다.** 요리/외식 모드 토글을 같은 탭 안에 두어, 비요리 사용자도 별도 앱처럼 느끼지 않게.
3. **도감은 항상 한 탭 거리.** 수집 동기를 상시 노출해 재방문 유도.

## 4. 핵심 사용자 여정 (User Journey)

세 개의 루프로 봅니다. **온보딩(1회) → 데일리 루프(핵심) → 위클리 루프(습관화).**

### 4.1 온보딩 — "마찰 최소화, 체중 입력 강요 금지"

```
가입 → 모찌와 첫 인사 → 취향 태그 3개만 (요리하는 편? / 알러지 / 비선호)
     → [선택] 냉장고 살짝 채우기  OR  "안 해먹어요" 선택
     → 첫 제안 즉시 제공 → "이거 좋아요!" 한 번 → 모찌 반응 → 끝
```
- **핵심 결정**: 온보딩에서 **체중/목표 입력을 요구하지 않는다.** (죄책감 제로 철학) 체중 기록은 나중에 마이 탭에서 *원하면* 시작.
- **분기점**: "요리하는 편?" 질문의 답으로 A/B 페르소나 경로가 갈리며, 첫 화면 제안 콘텐츠가 달라짐.

### 4.2 데일리 루프 — 리텐션의 심장

```
[푸시] "모찌가 오늘 메뉴 골라놨어요 🍳"
   → 앱 진입 → 오늘의 제안 확인
   → (먹음) '먹었어요' 체크  /  (취향) 좋아요·싫어요
   → 모찌 반응 (기뻐서 통통 튐) + 도감 카드 1장 획득 연출
   → 스트릭 +1
```
이 루프가 **하루 30초 안에 완결**되어야 합니다. 길어지면 초보자가 이탈합니다.

### 4.3 위클리 루프 — 습관 고착

```
[월] 새 주간 식단 자동 생성 ("이번 주 테마: 단백질 +α")
[주중] 재료 변동 시 식단 자동 적응 / 안 먹은 날은 다음 날로 재배치
[일] 다음 주 미리보기 + 장보기 리스트 자동 생성 → 좋아요/싫어요로 취향 학습
```

## 6. 데이터 시각화 — 체중 기록 (강박 제로 × 성취감 극대화)

> 전제: 첨부 예정인 `체중변화.xlsx`의 **월별·년도별 표/그래프 구조**를 앱에 옮기되, 숫자 강박을 빼고 모찌를 동행시킵니다.

### 6.1 시각화 5원칙

1. **추세 > 절대 수치.** Y축 눈금은 흐리게(또는 토글로 숨김). 화면의 주인공은 점 하나가 아니라 *부드럽게 흐르는 곡선*.
2. **빨강/초록 대비 금지.** 늘어도 비난하지 않는다. 모든 구간을 파스텔 단색(라벤더~민트 그라데이션) 영역 그래프로. → 체중이 올라간 날도 시각적으로 '실패'로 보이지 않음.
3. **일일 변동 숨기고 구간 평균.** 기본 뷰는 **주/월 평균선**. 매일 재는 압박(daily weigh-in pressure)을 구조적으로 제거. (원하면 '자세히'에서 일별 점 표시)
4. **목표선은 점선 + 부드럽게.** "여기까지 와야 해" 가 아니라 "이쯤 가고 있어요" 의 톤. 미달해도 경고 없음.
5. **빈 날은 공백이 아니라 휴식.** 기록 안 한 날 → 모찌가 "쉬어가도 괜찮아요 😴". 끊긴 그래프 = 실패라는 인상 차단.

### 6.2 모찌 활용 아이디어 (이게 핵심 차별점)

- **그래프 위를 걷는 모찌**: 체중 곡선 위에 모찌가 캐릭터로 서 있음. 추세가 내려가면 모찌가 *가벼워져 점프*, 정체기엔 *옆에서 응원하며 걸음*. → 숫자 대신 **캐릭터의 몸짓으로 진행도를 감정적으로 전달**.
- **"과거의 나" 비교를 모찌로**: before/after 체중 사진(부담스러움) 대신, **모찌의 성장 단계 비교**("3개월 전 모찌 vs 지금 모찌")로 성취를 시각화.
- **마일스톤 축하 연출**: 특정 구간 도달 시 모찌가 폭죽 터뜨리고 새 옷/테마 언락. 숫자 달성이 아니라 *이벤트*로 경험.
- **문장형 인사이트**: 그래프 하단에 "지난달보다 흐름이 좋아지고 있어요" 같은 **서술형 한 줄**. 모찌가 말풍선으로 전달.

### 6.3 표(Table) 뷰

- **월별 요약 테이블**: 월 / 평균 / 추세 화살표. 셀 배경은 **파스텔 그라데이션**(heatmap이되 부드럽게), 강한 색 금지.
- **년도별 토글**: xlsx의 연도 구조를 그대로 — 월별 ↔ 년도별 전환 탭.
- 표는 기본 숨김(마이 → '더보기'), *데이터를 보고 싶은 사람만* 진입.

## 7. 수집 시스템 고도화 (리텐션 엔진 디테일)

### 7.1 등급(희귀도) 체계

희귀도는 **재료 빈도·계절성·난이도**로 결정. 흔할수록 쉽게, 특별할수록 귀하게.

| 등급 | 재료 도감 예시 | 요리 도감 예시 | 색/연출 |
|---|---|---|---|
| ⚪ **일반(Common)** | 계란, 양파, 두부 | 계란 스크램블 | 기본 카드 |
| 🔵 **레어(Rare)** | 아스파라거스, 새우 | 단백질 볼 | 카드 테두리 반짝 |
| 🟣 **에픽(Epic)** | 트러플, 한우, 제철 해산물 | 마스터 레시피 | 홀로그램 효과 |
| 🌸 **한정(Seasonal/Event)** | 봄나물, 크리스마스 한정 | 시즌 테마 요리 | 기간 한정 뱃지 + FOMO |

### 7.2 컴플리트 보상 구조

수집을 **카테고리 → 시리즈 → 전체** 3단계로 묶어 단계별 보상.

- **카테고리 완성** (예: 채소 도감 100%) → 냉장고 테마 언락
- **시리즈 완성** (예: '15분 한 끼' 레시피 10종) → 모찌 특별 의상
- **시즌 완성** (예: 봄 한정 8종) → 한정 뱃지 + 명예의 전당 등재

### 7.3 "수집욕"을 자극하는 7가지 디테일

1. **실루엣 티저** — 미획득 카드를 실루엣+❓로 노출. "이건 뭘까?" 호기심 자극. (포켓몬 도감의 핵심 심리)
2. **진척 넛지** — "이 카테고리, 앞으로 **3개**만 더!" 게이지 옆 카피. *거의 다 왔다는 느낌*이 가장 강한 동기.
3. **히든 콤보** — 특정 재료 조합으로만 해금되는 **숨은 레시피**("두부+김치+계란 → 히든!"). 탐색의 재미.
4. **시즌 한정 FOMO(부드럽게)** — "이번 봄에만 모을 수 있어요". 단, 압박 톤 금지, 모찌가 살짝 귀띔하는 정도.
5. **모찌 플레이버 텍스트** — 각 카드에 캐릭터 코멘트("브로콜리, 모찌가 제일 좋아하는 나무 🌳"). 수집물에 *정서적 애착* 부여.
6. **'오늘의 발견' 연출** — 새 재료·레시피 첫 등록 시 특별 카드 오픈 애니메이션 + 효과음. 매 첫 획득을 *작은 이벤트*로.
7. **도감 공유 카드** — 내 컬렉션을 예쁜 이미지로 내보내 친구에게 자랑. (소셜 바이럴 + 무료 마케팅 채널)

## 8. ⭐ [핵심 과제] 비(非)요리 사용자 포용 방안

> **문제 재정의**: 앱이 '냉장고 비우기'에 묶이면 페르소나 B(배달·외식·간편식 위주)가 이탈한다.
> **해결 원칙**: 앱의 3대 루프(**제안 · 기록 · 수집**)를 비요리 사용자도 **전부** 경험하게 만든다. 아래 3가지 아이디어는 각각 하나의 루프를 비요리 맥락으로 확장한 것입니다.

### 💡 아이디어 1 — "외식/배달 추천 모드" (제안 루프 확장)

**핵심 통찰**: "오늘 뭐 먹지 대신 정해주기"라는 USP는 *요리 안 하는 사람에게 오히려 더 절실하다.* "오늘 또 뭐 시키지"가 매일의 고통이니까.

- `식단` 탭의 `ModeToggle`로 **[외식/배달 모드]** 진입. 추천 로직(단백질·포만감·가벼움 랭킹)을 **그대로** 메뉴 추천에 적용.
- 모찌: "오늘은 닭가슴살 샐러드 어때요? 🥗" → 위치 기반으로 **주변 매장/배달 카테고리** 제안.
- 배달앱 딥링크(배민·쿠팡이츠 등) 또는 제휴로 연결. *주문은 외부에서, 결정과 기록은 모찌에서.*
- 메뉴 선택 → `먹었어요` 체크 → 모찌 성장·도감 적립 **동일 적용**.
- **결과**: 냉장고가 비어 있어도 **제안 → 선택 → 기록 → 보상** 루프가 완결된다.

### 💡 아이디어 2 — "간편식·편의점 도감" (수집 루프 확장)

**핵심 통찰**: 비요리 사용자에게도 *모을 거리*를 줘야 도감 훅이 작동한다. 안 만들어도 수집은 할 수 있다.

- `도감`에 **간편식 도감** 탭 추가: 편의점 PB·밀키트·냉동식품·외식 메뉴를 수집 대상으로.
- **편의점 조합 추천**: "닭가슴살 + 그릭요거트 + 바나나" 같은 *편의점에서 완성하는 균형 한 끼* 제안 (밀프렙 못 하는 사람용 현실적 대안).
- **바코드 스캔으로 간편식 등록** → 영양 자동 인식 → 도감 카드 획득. (이미 기획된 바코드 기능을 비요리 맥락으로 재활용 — 추가 개발 부담 적음)
- 브랜드 제휴 시 한정 카드 가능(예: "○○ 신상 도시락 도감").
- **결과**: '냉장고 비우기'가 아니라 '**오늘 먹은 것 수집하기**'로 프레임이 확장된다.

### 💡 아이디어 3 — "사진 한 장 식사 기록" (기록 루프 확장)

**핵심 통찰**: 비요리 사용자의 진짜 진입장벽은 *입력 자체*다. 요리 과정이 없으니 기록할 이유도 약하다 → **입력을 0초에 가깝게**.

- 먹은 음식(외식·배달) **사진 1장** → AI가 메뉴 인식 → 자동 기록 + 도감 등록.
- **정확한 칼로리 불필요** (죄책감 제로 철학과 일치). "오늘 든든하게 드셨네요 😊" 정도의 *느슨한 피드백*.
- 영수증/카드 내역 스캔으로 외식 기록 보조.
- **부드러운 밸런싱**: 며칠 외식이 이어지면 모찌가 "내일은 가벼운 거 어때요?" 라고 *제안*(경고 아님)하며 1번 아이디어로 자연 순환.
- **결과**: 요리 없이 **먹기 → 찍기 → 기록 → 모찌 칭찬**만으로 데일리 루프가 돈다.

> **세 아이디어의 시너지**: 1번이 *결정*을, 3번이 *기록*을, 2번이 *수집*을 담당하며 비요리 사용자도 요리 사용자와 **동일한 리텐션 루프**에 올라탑니다. 냉장고는 여러 입력 모드 중 하나일 뿐이라는 8장 도입부의 재정의가 여기서 완성됩니다.

## 9. MVP 범위 & 개발 우선순위 (프론트엔드 관점)

> 전부 만들면 출시가 늦어집니다. **'제안 → 기록 → 수집' 최소 루프**를 P0로 먼저 검증하길 권합니다. (포트폴리오/사이드 프로젝트라면 P0만으로도 완결성 있는 데모가 됩니다.)

| 우선순위 | 기능 | 이유 |
|---|---|---|
| **P0 (MVP)** | 홈(모찌 상태)·수동 재료 입력·레시피 추천(매칭률)·먹었어요 체크·기본 도감·스트릭 | 핵심 루프 1회전 검증. 이것만 돌아도 제품의 정체성이 선다 |
| **P1** | 외식/배달 모드(8장 1번)·사진 기록(8장 3번)·주간 식단 캘린더·체중 그래프(모찌 동행) | 타겟 확장 + 습관화. 비요리 사용자 유입의 핵심 |
| **P2** | 냉장고 사진 AI 인식·영수증/바코드·간편식 도감·등급/희귀도 고도화·도감 공유 | 정확도·수집 깊이·바이럴. 데이터 쌓인 뒤 고도화 |

**프론트 구현 메모 (참고)**
- 말랑 인터랙션(`젤리 버튼`, 모찌 표정 전환)은 **Framer Motion**(spring 애니메이션)이 적합. `whileTap={{ scale: 0.92 }}` 류로 손쉽게 구현.
- 모찌 캐릭터는 상태가 많으므로 **Lottie**(JSON 애니메이션) 또는 상태별 SVG 스프라이트 권장. 초기엔 정적 SVG + transition으로 시작해도 충분.
- 체중 그래프는 **Recharts**의 `AreaChart` + 커스텀 `dot`으로 '모찌 걷기' 구현 가능. 곡선은 `type="monotone"`로 부드럽게.
- 죄책감 제로 = **컬러 토큰 설계가 핵심**. 디자인 토큰에 빨강 계열을 *경고용으로조차 넣지 않는 것*을 규칙으로.

## 11. v1.1 보완 — 효과·개인화·기록 확장 (2026-06-30 논의)

> P0 핵심 루프(제안→기록→수집)를 구현·배포한 뒤 나온 제품 질문 4가지를 반영한다.
> 네 가지는 하나로 수렴한다: **"앱이 실제로 다이어트 효과를 주려면 약간의 *가이드 지능*이 필요하고, 그건 죄책감 제로와 양립한다 — 벌이 아니라 (스마트 제안 + 긍정 프레이밍 + opt-in 개인화)로 전달할 때."**

### 11.1 핵심 재확인 — 효과는 "제안(미래)"에서, 죄책감 제로는 "기록(과거)"에서

리스크 10장이 인지한 "죄책감 제로 vs 효과" 긴장의 해법을 명문화한다.

- **이미 먹은 것(과거)엔 절대 벌을 주지 않는다.** ("라면? 괜찮아요" — 죄책감 제로 유지)
- **다음 선택(미래)은 똑똑하게 유도한다.** Decision-as-a-Service의 힘 = 모찌가 *좋은 기본값*을 골라주는 것. → "맨날 고칼로리 + 모찌 응원"으로 효과를 못 보는 실패를 여기서 막는다.
- **숫자 없는 부드러운 가이드**: 영양 정보는 "1200 kcal ❌"가 아니라 **"오늘 단백질 잘 챙겼어요 / 가벼운 하루였어요"** 같은 긍정 프레이밍으로만 노출. (영양 수치 자체는 서버 전용, 화면 비노출 — 불변 #2)
- ⚠️ **과교정 금지**: 여기서 엄격한 칼로리 통제 앱으로 가면 USP를 배신한다. "유도하되 압박하지 않는" 균형이 제품의 예술.

### 11.2 여러 끼니 기록 (아침·점심·저녁·간식)

- 데일리 루프(4.2)의 "하루 하나" 프레이밍을 확장: `MealRecord`에 **끼니 슬롯**(breakfast/lunch/dinner/snack)을 두어 여러 끼니를 기록.
- ⚠️ **가볍게 유지**: 3끼 강제 로깅은 MyFitnessPal의 귀찮음이 된다. **탭 한 번 + 시간대로 슬롯 자동추정**. 30초 완결 원칙(4.2)은 지킨다.
- 하루 전체가 보여야 11.5 밸런싱이 가능하므로 효과의 전제.

### 11.3 사용자 정의 요리 (내 레시피)

- 냉장고 재료로 만드는 *자기만의 요리*를 등록 가능하게. `Recipe`에 소유자 개념(비면 시드, 있으면 내 요리)을 두어 **요리 모드 추천에 내 요리도 포함**.
- 기존 루프에 그대로 플러그인: 내 요리 '먹었어요' → 모찌 반응 + 도감 적립. 내 요리가 내 도감에 쌓여 애착·리텐션↑.

### 11.4 opt-in 개인 프로필 (나이·성별·키·활동량)

- 맞춤 관리엔 개인 정보가 필요하지만(BMR/TDEE), 온보딩(4.1)에서 강요하면 죄책감 제로·저마찰 원칙과 충돌.
- **해법 = opt-in**: 마이 탭에서 *원하는 사람만* 나이/성별/키/활동량 입력 → 개인화된 **부드러운** 가이드(하드 캡 X, "오늘 이 정도면 딱 좋아요"). 저마찰 온보딩은 유지.
- 이 프로필이 11.5 밸런싱과 11.1 가이드를 개인별로 의미있게 만든다.

### 11.5 트렌드 기반 밸런싱 넛지 (8장 아이디어3 구체화)

- 서버측 영양 데이터로 **최근 식사 추세를 추적** → 며칠 무거우면 모찌가 *가벼운 쪽을 제안*(경고 아님). 응원 톤 유지, 제안이 교정.
- 체중 그래프(6장, 구현됨)를 **정직한 거울**로: 효과가 안 보이면 그래프가 보여주고(비난 X) 모찌가 조정을 제안.
